# Plan 01-03: Nested OU Structure via OpenTofu + Terragrunt

**Phase:** 01-Foundation
**Requirements:** FOUND-03
**Goal:** AWS Organizations shows the target nested OU structure — Root → chain-vote-ai → {dev, staging, prod}; Root → chain-vote-voting → {dev, staging, prod} — created via OpenTofu resources in a Terragrunt unit with explicit parent-before-child dependency ordering.

## Pre-conditions

- [ ] Plan 01-01 is complete: state backend (S3 + DynamoDB) exists and root `terragrunt.hcl` is wired
- [ ] Plan 01-02 is complete: CT landing zone is deployed and `management/ct-bootstrap/` unit is initialized; `terragrunt output` returns valid account IDs
- [ ] The caller identity has `organizations:CreateOrganizationalUnit`, `organizations:ListRoots`, and `organizations:ListOrganizationalUnitsForParent` permissions
- [ ] CT landing zone shows status **Ready** in the CT console (CT must be live before OU structure is created, since CT will register OUs and can conflict with pre-existing OUs if creation order is wrong)

## Tasks

### Task 1: Write the `management/ou-structure/` Terragrunt unit

**Files:**
- `management/ou-structure/main.tf` — `aws_organizations_organizational_unit` resources for the 8-OU tree, ordered by explicit `depends_on`
- `management/ou-structure/outputs.tf` — exports OU IDs for each of the 8 OUs (consumed by Phase 3 account vending)
- `management/ou-structure/terragrunt.hcl` — child unit config with dependency on `management/ct-bootstrap/`

**Steps:**

1. Create `management/ou-structure/terragrunt.hcl`:

   ```hcl
   # Inherits terraform_binary = "tofu" and remote_state from root terragrunt.hcl.
   include "root" {
     path = find_in_parent_folders()
   }

   # Depend on ct-bootstrap to ensure CT is live and root OU ID is available.
   # This also enforces the Plan 01-02 → Plan 01-03 ordering at the Terragrunt level.
   dependency "ct_bootstrap" {
     config_path = "../ct-bootstrap"
     mock_outputs = {
       management_account_id = "000000000000"
     }
     mock_outputs_allowed_terraform_commands = ["validate", "plan"]
   }
   ```

2. Create `management/ou-structure/main.tf`:

   **Critical ordering constraint (Pitfall 5):** CT requires parent OUs to be created before child OUs. OpenTofu processes `depends_on` within a plan before ordering resource creation. The two parent OUs (`chain-vote-ai`, `chain-vote-voting`) are created first, each depending on the root. The six child OUs each have explicit `depends_on` referencing their respective parent OU resource.

   **Step 1 — Read the Organizations root ID:**
   ```hcl
   data "aws_organizations_organization" "current" {}
   ```

   **Step 2 — Parent OUs (under Root):**
   ```hcl
   resource "aws_organizations_organizational_unit" "chain_vote_ai" {
     name      = "chain-vote-ai"
     parent_id = data.aws_organizations_organization.current.roots[0].id

     tags = {
       ManagedBy = "opentofu"
       Phase     = "01-foundation"
     }
   }

   resource "aws_organizations_organizational_unit" "chain_vote_voting" {
     name      = "chain-vote-voting"
     parent_id = data.aws_organizations_organization.current.roots[0].id

     tags = {
       ManagedBy = "opentofu"
       Phase     = "01-foundation"
     }
   }
   ```

   **Step 3 — Child OUs under chain-vote-ai:**
   Each child uses `parent_id = aws_organizations_organizational_unit.chain_vote_ai.id`. OpenTofu's dependency graph implicitly ensures `chain_vote_ai` is created first when `parent_id` references it — no explicit `depends_on` needed for the parent-child relationship (the resource attribute reference IS the dependency).

   ```hcl
   resource "aws_organizations_organizational_unit" "chain_vote_ai_dev" {
     name      = "dev"
     parent_id = aws_organizations_organizational_unit.chain_vote_ai.id

     tags = {
       ManagedBy   = "opentofu"
       Phase       = "01-foundation"
       Environment = "dev"
       Workload    = "chain-vote-ai"
     }
   }

   resource "aws_organizations_organizational_unit" "chain_vote_ai_staging" {
     name      = "staging"
     parent_id = aws_organizations_organizational_unit.chain_vote_ai.id

     tags = {
       ManagedBy   = "opentofu"
       Phase       = "01-foundation"
       Environment = "staging"
       Workload    = "chain-vote-ai"
     }
   }

   resource "aws_organizations_organizational_unit" "chain_vote_ai_prod" {
     name      = "prod"
     parent_id = aws_organizations_organizational_unit.chain_vote_ai.id

     tags = {
       ManagedBy   = "opentofu"
       Phase       = "01-foundation"
       Environment = "prod"
       Workload    = "chain-vote-ai"
     }
   }
   ```

   **Step 4 — Child OUs under chain-vote-voting:**
   ```hcl
   resource "aws_organizations_organizational_unit" "chain_vote_voting_dev" {
     name      = "dev"
     parent_id = aws_organizations_organizational_unit.chain_vote_voting.id

     tags = {
       ManagedBy   = "opentofu"
       Phase       = "01-foundation"
       Environment = "dev"
       Workload    = "chain-vote-voting"
     }
   }

   resource "aws_organizations_organizational_unit" "chain_vote_voting_staging" {
     name      = "staging"
     parent_id = aws_organizations_organizational_unit.chain_vote_voting.id

     tags = {
       ManagedBy   = "opentofu"
       Phase       = "01-foundation"
       Environment = "staging"
       Workload    = "chain-vote-voting"
     }
   }

   resource "aws_organizations_organizational_unit" "chain_vote_voting_prod" {
     name      = "prod"
     parent_id = aws_organizations_organizational_unit.chain_vote_voting.id

     tags = {
       ManagedBy   = "opentofu"
       Phase       = "01-foundation"
       Environment = "prod"
       Workload    = "chain-vote-voting"
     }
   }
   ```

   **Why resource attribute references are sufficient for ordering:** When `parent_id` is set to `aws_organizations_organizational_unit.chain_vote_ai.id`, OpenTofu automatically creates `chain_vote_ai` before any resource that references it. Explicit `depends_on` blocks are not needed and would be redundant here. Do NOT use `run-all` for this unit — apply it as a single `terragrunt apply` call from `management/ou-structure/`. All 8 resources are in one state file and OpenTofu will sequence them correctly.

### Task 2: Register OUs with CT and verify the full structure

**Files:**
- `docs/runbooks/02-ou-ct-registration.md` — operator runbook for registering the 8 OUs with CT after OpenTofu creates them

**Steps:**

1. Create `docs/runbooks/02-ou-ct-registration.md`.

   **Section: Why OUs must be registered separately from creation**
   Explain: `aws_organizations_organizational_unit` creates the OU in AWS Organizations, but CT does not automatically govern new OUs. Each OU must be explicitly registered with CT. CT enforces parent-before-child registration order — the same constraint that applies to OU creation.

   **Section: Capture OU IDs from Terragrunt outputs**
   ```bash
   cd management/ou-structure
   AI_PARENT_OU=$(terragrunt output -raw chain_vote_ai_ou_id)
   VOTING_PARENT_OU=$(terragrunt output -raw chain_vote_voting_ou_id)
   AI_DEV_OU=$(terragrunt output -raw chain_vote_ai_dev_ou_id)
   AI_STAGING_OU=$(terragrunt output -raw chain_vote_ai_staging_ou_id)
   AI_PROD_OU=$(terragrunt output -raw chain_vote_ai_prod_ou_id)
   VOTING_DEV_OU=$(terragrunt output -raw chain_vote_voting_dev_ou_id)
   VOTING_STAGING_OU=$(terragrunt output -raw chain_vote_voting_staging_ou_id)
   VOTING_PROD_OU=$(terragrunt output -raw chain_vote_voting_prod_ou_id)
   cd ../..
   ```

   **Section: Register parent OUs first (strictly sequential)**
   CT requires each registration to COMPLETE before the next starts. Do not batch these.

   Register `chain-vote-ai` (wait for SUCCEEDED before proceeding):
   ```bash
   aws controltower enable-baseline \
     --baseline-identifier arn:aws:controltower:us-east-1::baseline/AWSControlTowerBaseline \
     --baseline-version "1.0" \
     --target-identifier "${AI_PARENT_OU}"
   # Wait: check CT console or poll:
   # aws controltower get-baseline-operation --operation-identifier <id>
   # until Status == "SUCCEEDED"
   ```

   Register `chain-vote-voting` (after `chain-vote-ai` SUCCEEDED):
   ```bash
   aws controltower enable-baseline \
     --baseline-identifier arn:aws:controltower:us-east-1::baseline/AWSControlTowerBaseline \
     --baseline-version "1.0" \
     --target-identifier "${VOTING_PARENT_OU}"
   # Wait for SUCCEEDED
   ```

   Note: The `enable-baseline` API call is the CT API for OU registration. In the CT console, this is the "Register OU" workflow. The `AWSControlTowerBaseline` ARN is the standard CT baseline; verify it for your region using `aws controltower list-baselines`.

   **Section: Register child OUs (after parent SUCCEEDED)**
   Once each parent OU registration is SUCCEEDED, register its children. Registrations within the same parent CAN proceed in parallel (they have independent parents). Registrations across parent OUs are independent.

   ```bash
   # chain-vote-ai children (run after chain-vote-ai parent SUCCEEDED)
   for OU_ID in "${AI_DEV_OU}" "${AI_STAGING_OU}" "${AI_PROD_OU}"; do
     aws controltower enable-baseline \
       --baseline-identifier arn:aws:controltower:us-east-1::baseline/AWSControlTowerBaseline \
       --baseline-version "1.0" \
       --target-identifier "${OU_ID}"
     echo "Submitted registration for ${OU_ID} — wait for SUCCEEDED in CT console before next step"
     sleep 5
   done
   ```

   Repeat for `chain-vote-voting` children after its parent SUCCEEDED.

   **Section: Verify final structure**
   After all 8 OUs are registered (CT status SUCCEEDED for each):
   ```bash
   # Verify OU tree in Organizations
   aws organizations list-organizational-units-for-parent \
     --parent-id $(aws organizations list-roots --query 'Roots[0].Id' --output text) \
     --query 'OrganizationalUnits[*].{Name:Name,Id:Id}' --output table

   # Verify CT enrollment status for each parent OU
   aws controltower list-enabled-baselines \
     --query 'enabledBaselines[*].{Target:targetIdentifier,Status:statusSummary.status}' \
     --output table

   # Expected: chain-vote-ai and chain-vote-voting show SUCCEEDED
   ```

   **Section: Troubleshooting**
   - If an OU registration shows FAILED: check CloudFormation StackSet operations in the management account. The error is usually a pre-existing Config recorder in a target account.
   - If registration is stuck IN_PROGRESS for > 20 minutes: check CT console for "Setup in progress" banner; another CT operation may be running concurrently.
   - Do NOT attempt to register the `Security` OU or the Root — these are CT-managed automatically.

## Threat Model

| Risk | Mitigation |
|------|-----------|
| OU creation succeeds but CT registration fails because parent OU is not yet registered | Runbook enforces strict sequential registration: parent SUCCEEDED before child registration is submitted; CT API will return `ConstraintViolationException` if parent not ready, prompting retry |
| `run-all apply` accidentally used across `management/` triggering parallel OU creation and CT registration | Runbook explicitly prohibits `run-all`; each unit (`ct-bootstrap`, `ou-structure`) is applied individually with `terragrunt apply` from the unit directory |
| OU naming collision with CT's auto-created `Security` OU | Plan creates only `chain-vote-ai` and `chain-vote-voting` under Root; `Security` OU is CT-managed and untouched |
| Someone deletes an OU via console after IaC creates it | `prevent_destroy = true` is intentionally NOT set (OUs must be deletable if a workload is retired); drift is detectable via `terragrunt plan` which will show the resource as missing and plan to recreate it |
| CT registration of child OU fails because parent registration stalled | Runbook includes troubleshooting section pointing to CloudFormation StackSet events; `enable-baseline` operation ID returned on submit can be polled via `get-baseline-operation` |
| Duplicate OU names at same parent level (e.g., two `dev` OUs under `chain-vote-ai`) | OpenTofu creates a single `aws_organizations_organizational_unit` resource per OU; no duplicates possible from IaC; manual duplicates in Organizations console would cause `terragrunt plan` to show drift |
