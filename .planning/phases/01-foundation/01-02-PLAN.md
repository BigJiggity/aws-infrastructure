# Plan 01-02: CT Landing Zone Runbook + IaC Wrapper

**Phase:** 01-Foundation
**Requirements:** FOUND-02
**Goal:** CT landing zone is deployed in us-east-1 (via console), and an OpenTofu data source unit at `management/ct-bootstrap/` captures its outputs (Log Archive account ID, Audit account ID, landing zone ARN) for downstream use.

## Pre-conditions

- [ ] Plan 01-01 is complete: `scripts/preflight.sh` passes all 5 checks with exit 0 (no CT enrollment blockers)
- [ ] State backend is bootstrapped: S3 bucket and DynamoDB table exist (Plan 01-01 Task 2)
- [ ] Root `terragrunt.hcl` exists with `terraform_binary = "tofu"` and `remote_state` config
- [ ] Operator has management account console access with AdministratorAccess
- [ ] Operator has chosen email addresses for the Log Archive and Audit accounts (must be globally unique across AWS)

## Tasks

### Task 1: Write the CT landing zone runbook

**Files:**
- `docs/runbooks/01-ct-landing-zone.md` — step-by-step operator runbook for deploying CT via the AWS console (per D-04)

**Steps:**

1. Create directory `docs/runbooks/`.

2. Create `docs/runbooks/01-ct-landing-zone.md` with these sections:

   **Section: Overview**
   State explicitly: CT is deployed via the AWS console, not via `tofu apply`. CT self-manages its own CloudFormation StackSets internally and conflicts with external IaC if you attempt to manage CT via Terraform resources. This runbook must be followed in order. Estimated time: 45–60 minutes for CT setup; 15–30 minutes for IaC wrapper.

   **Section: Pre-flight gate**
   Instruction: Run `scripts/preflight.sh` from the repo root. Do NOT proceed if exit code is non-zero. All 5 checks must pass.
   ```bash
   ./scripts/preflight.sh
   # Expected: 5 PASS lines, exit 0
   ```

   **Section: State backend gate**
   Instruction: Verify state backend exists before any Terragrunt commands.
   ```bash
   ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
   aws s3api head-bucket --bucket "chain-vote-tofu-state-${ACCOUNT_ID}"
   aws dynamodb describe-table --table-name chain-vote-tofu-locks --region us-east-1
   # Both commands must succeed (exit 0). If not, run: ./scripts/bootstrap-state.sh
   ```

   **Section: Enable CT Organizations trusted access**
   Instruction: Before opening the CT console, ensure trusted access is enabled for all required service principals. Run:
   ```bash
   aws organizations enable-aws-service-access --service-principal controltower.amazonaws.com
   aws organizations enable-aws-service-access --service-principal config.amazonaws.com
   aws organizations enable-aws-service-access --service-principal cloudtrail.amazonaws.com
   aws organizations enable-aws-service-access --service-principal sso.amazonaws.com
   ```
   Idempotent — safe if already enabled.

   **Section: Deploy CT landing zone via AWS console**
   Step-by-step with exact console navigation:
   1. Sign in to the AWS management account console. Navigate to: **AWS Control Tower** (search in the top bar).
   2. Click **Set up landing zone**.
   3. **Home Region:** Select `us-east-1 (US East - N. Virginia)`.
   4. **Log archive account:** Enter a unique email and account name `log-archive`. Record the email used.
   5. **Audit account:** Enter a unique email and account name `audit`. Record the email used.
   6. **Foundational OU:** Accept the default `Security` OU name.
   7. **Additional OU:** Add `Sandbox` (CT requires at least one additional OU at setup; it can be deleted after setup is complete — see Pitfall 5 note on OU registration ordering).
   8. Review the preview. CT will create: a CloudTrail organization trail, AWS Config aggregator, two new accounts, and IAM Identity Center.
   9. Click **Set up landing zone**. Estimated time: 30–60 minutes.
   10. Monitor progress in the CT console. Do not navigate away. Setup is complete when status shows **Ready**.

   **Section: Post-setup verification**
   After CT shows Ready:
   ```bash
   # Verify the landing zone ARN
   aws controltower list-landing-zones --query 'landingZones[0].arn'

   # Verify Log Archive and Audit account IDs
   aws organizations list-accounts --query 'Accounts[?Status==`ACTIVE`].[Name,Id,Email]' --output table
   # Look for accounts named 'log-archive' and 'audit'

   # Verify Organizations trusted access is on for CT
   aws organizations list-aws-service-access-for-organization \
     --query 'EnabledServicePrincipals[*].ServicePrincipal' --output json
   ```

   **Section: Record account IDs**
   Instruction: Capture the three account IDs for use in Plan 01-02 Task 2 and Phase 2 (AFT module inputs):
   ```bash
   MGMT_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
   LOG_ARCHIVE_ID=$(aws organizations list-accounts \
     --query 'Accounts[?Name==`log-archive`].Id' --output text)
   AUDIT_ID=$(aws organizations list-accounts \
     --query 'Accounts[?Name==`audit`].Id' --output text)

   echo "Management account : ${MGMT_ACCOUNT_ID}"
   echo "Log Archive account: ${LOG_ARCHIVE_ID}"
   echo "Audit account      : ${AUDIT_ID}"
   ```
   Save these values. They are required inputs for `management/ct-bootstrap/variables.tf` (Task 2) and the AFT module in Phase 2.

   **Section: Known pitfalls**
   Reference PITFALLS.md. Key warnings to include inline:
   - Pre-existing Config recorders block setup (Pitfall 1) — pre-flight script catches this.
   - Pre-existing IAM roles block enrollment (Pitfall 2) — pre-flight script catches this.
   - CT console changes after IaC takeover create drift (Pitfall 7) — establish no-console-changes policy now.
   - Do NOT register the `Security` OU in CT; it is auto-managed. Do NOT modify SCPs on CT-managed OUs.

**Verification:**
- [ ] `docs/runbooks/01-ct-landing-zone.md` exists with all sections listed above
- [ ] The runbook includes exact CLI commands for pre-flight gate, trusted access enablement, and account ID capture
- [ ] The runbook explicitly states CT is NOT managed via `tofu apply`

### Task 2: Write the `management/ct-bootstrap/` OpenTofu data source unit

**Files:**
- `management/ct-bootstrap/main.tf` — reads CT landing zone and accounts via data sources; exports outputs
- `management/ct-bootstrap/variables.tf` — input vars for account IDs (populated after CT console setup)
- `management/ct-bootstrap/outputs.tf` — exports landing_zone_arn, log_archive_account_id, audit_account_id
- `management/ct-bootstrap/terragrunt.hcl` — child unit config inheriting root remote_state

**Steps:**

1. Create `management/ct-bootstrap/terragrunt.hcl`:
   ```hcl
   # Inherits terraform_binary = "tofu" and remote_state from root terragrunt.hcl.
   include "root" {
     path = find_in_parent_folders()
   }
   ```
   No additional configuration needed — all state and provider config comes from the root include.

2. Create `management/ct-bootstrap/variables.tf`:
   ```hcl
   variable "management_account_id" {
     description = "AWS account ID of the CT management account. Set after CT setup."
     type        = string
   }

   variable "log_archive_account_id" {
     description = "AWS account ID of the CT-managed Log Archive account. Set after CT console setup."
     type        = string
   }

   variable "audit_account_id" {
     description = "AWS account ID of the CT-managed Audit account. Set after CT console setup."
     type        = string
   }
   ```

3. Create `management/ct-bootstrap/main.tf`:

   Purpose: Read the CT landing zone and AWS Organization data via data sources. This unit does NOT create any resources — it is purely a data source / output exporter. IaC does not manage CT itself (per D-03).

   ```hcl
   # Data source: CT landing zone
   # Reads the live landing zone to capture its ARN.
   # aws_controltower_landing_zone data source requires provider >= 5.27.0 (included in >= 6.0.0).
   data "aws_controltower_landing_zones" "current" {}

   # Data source: AWS Organization
   data "aws_organizations_organization" "current" {}

   # Data source: Log Archive account
   data "aws_organizations_account" "log_archive" {
     id = var.log_archive_account_id
   }

   # Data source: Audit account
   data "aws_organizations_account" "audit" {
     id = var.audit_account_id
   }
   ```

   Note on `aws_controltower_landing_zones` (plural): The provider resource is `aws_controltower_landing_zones` (data source, returns a list). Access the first element's ARN as `data.aws_controltower_landing_zones.current.arns[0]`. Verify this data source name is correct for `hashicorp/aws >= 6.0.0` before running `tofu init`. If the data source does not exist in the provider, use the AWS CLI output approach: read the ARN from a local file written by the runbook's capture step, and expose it as a `locals {}` block instead.

4. Create `management/ct-bootstrap/outputs.tf`:
   ```hcl
   output "landing_zone_arn" {
     description = "ARN of the CT landing zone. Consumed by Phase 2 AFT module."
     value       = try(data.aws_controltower_landing_zones.current.arns[0], "NOT_YET_DEPLOYED")
   }

   output "log_archive_account_id" {
     description = "Account ID of the CT-managed Log Archive account."
     value       = data.aws_organizations_account.log_archive.id
   }

   output "audit_account_id" {
     description = "Account ID of the CT-managed Audit account."
     value       = data.aws_organizations_account.audit.id
   }

   output "management_account_id" {
     description = "Account ID of the CT management account."
     value       = data.aws_organizations_organization.current.master_account_id
   }
   ```

5. Create `management/ct-bootstrap/terraform.tfvars.example`:
   ```hcl
   # Copy to terraform.tfvars and fill in values after CT console setup.
   # These values come from the account ID capture step in the runbook.
   management_account_id  = "REPLACE_WITH_MGMT_ACCOUNT_ID"
   log_archive_account_id = "REPLACE_WITH_LOG_ARCHIVE_ACCOUNT_ID"
   audit_account_id       = "REPLACE_WITH_AUDIT_ACCOUNT_ID"
   ```
   Add `management/ct-bootstrap/terraform.tfvars` to `.gitignore` (account IDs are not secrets but the tfvars pattern keeps them out of git for clarity).

6. Update `.gitignore` at repo root to include:
   ```
   # Terragrunt-generated files
   **/.terragrunt-cache/
   **/.terraform/
   **/backend.tf
   **/provider.tf
   **/versions.tf
   # Local tfvars with account IDs
   **/terraform.tfvars
   # State files (should never be committed)
   **/*.tfstate
   **/*.tfstate.backup
   ```

## Threat Model

| Risk | Mitigation |
|------|-----------|
| CT console deployment blocked by pre-existing Config recorders or IAM roles | Pre-flight script (Plan 01-01 Task 1) hard-blocks before operator reaches this plan |
| Operator records wrong account IDs in tfvars | Runbook includes CLI verification commands to confirm IDs match account names; `aws_organizations_account` data source will fail to plan if an invalid account ID is provided |
| `aws_controltower_landing_zones` data source unavailable in provider version | `try()` guard returns `"NOT_YET_DEPLOYED"` instead of erroring; CI/CD is not blocked, but landing_zone_arn output will be a placeholder until CT is live |
| tfvars with account IDs committed to git | `.gitignore` excludes `**/terraform.tfvars`; `terraform.tfvars.example` is committed instead |
| IaC accidentally manages CT resources (violates D-03) | `management/ct-bootstrap/main.tf` contains only `data` blocks, no `resource` blocks; code reviewer must enforce this |
| CT console change after IaC takeover creates drift (Pitfall 7) | Runbook explicitly states no-console-changes policy; CT drift detection via CT console "Drift detected" banner |
